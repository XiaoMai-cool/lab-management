> 审查日期：2026-04-05

## 一、🔴 会导致数据错误或流程走不通的问题

### 1. 库存操作没有原子性保护——竞态条件导致库存数据错乱

**涉及文件：**
- `src/pages/supplies/ReservationReview.tsx:218-277`（审批扣库存）
- `src/pages/supplies/SupplyReturn.tsx:72-109`（归还加库存）
- `src/pages/supplies/ReservationReview.tsx:71-119`（撤回审批恢复库存）
- `src/pages/reagents/ChemicalWarnings.tsx:121-125`（药品到货更新库存）

**问题：** 所有库存变更都是「两步操作」——先调 `adjust_stock` 改库存，再更新记录状态，这两步不在一个事务里。而且更新记录时没有加 `WHERE status = 'pending'` 这样的乐观锁条件。

**后果（举例）：**
- 两个管理员同时批准同一条申领 → 库存被扣两次
- 用户点「归还」的同时管理员也在处理同一条归还 → 库存被加两次
- 管理员连续点两次「撤回审批」→ 库存被恢复两次
- 药品到货时用的是直接覆盖 `update({ stock: newStock })` 而非 delta 更新，两人同时操作会互相覆盖

**谁会遇到：** 管理员、任何操作库存的人，尤其在多人同时在线时
**修复难度：** 中等——需要写一个 PostgreSQL 函数，在一个事务内完成「检查状态 + 调整库存 + 更新记录」

---

### 2. `adjust_stock` 函数不在版本控制中，且数据库没有 `stock >= 0` 约束

**涉及文件：** `supabase/` 目录下所有 SQL 文件

**问题：**
- `adjust_stock` 这个 RPC 函数被 4 个前端文件调用，但在所有 SQL 迁移文件中找不到定义——它可能是直接在 Supabase 控制台手动创建的
- `supplies.stock` 和 `chemicals.stock` 字段没有 `CHECK (stock >= 0)` 约束

**后果：** 库存可以变成负数，数据库不会阻止；如果重建数据库，这个函数会丢失
**谁会遇到：** 开发者重建环境时；任何导致库存过度扣减的并发场景
**修复难度：** 简单——加 CHECK 约束 + 把函数写进迁移文件

---

### 3. Profile 加载失败时，所有权限检查被跳过

**文件：** `src/components/ProtectedRoute.tsx:36-49`

**问题：** 当 `profile` 为 `null`（网络不好没加载出来），代码注释写着「don't redirect」，直接放行。这意味着任何登录用户在 profile 加载失败时，可以访问 `/admin`、`/admin/members`、`/admin/export` 等所有管理员页面。

**后果：** 学生可能访问到管理员功能，包括人员管理、数据导出
**谁会遇到：** 网络不稳定的用户，或者故意拦截 profile 请求的人
**修复难度：** 简单——profile 为 null 时显示加载中或错误页面，不要放行

### 4. 删除耗材时，关联数据被级联删除但确认弹窗说了假话

**文件：** `src/pages/admin/SupplyManage.tsx:167-185, 536-538`

**问题：**
- 数据库设置了 `ON DELETE CASCADE`，删除耗材会连带删除所有预约记录和借用记录
- 但删除确认弹窗写的是「删除后相关预约记录将不受影响」——这是错的
- 已批准的预约（库存已扣减）被删除后，库存永远不会恢复
- 正在借用中的记录被删除后，借出的物品再也无法通过系统归还

**后果：** 管理员被误导，以为删除是安全的；实际上会丢失借用追踪数据，库存账目永久不平
**谁会遇到：** 管理员清理不再使用的耗材时
**修复难度：** 中等——应在删除前检查是否有活跃的预约/借用，有则阻止删除或先处理

---

### 5. 用户撤回申领与管理员审批的竞态条件

**文件：** `src/pages/supplies/MyReservations.tsx:94-109`

**问题：** 用户撤回 pending 申领时，只按 `id` 删除，没加 `WHERE status = 'pending'`。如果管理员刚好在这一刻批准了（库存已扣减），用户的删除操作仍会成功（CASCADE 删除），但扣掉的库存永远不会恢复。

**后果：** 库存永久减少，但没有对应的实际领用
**谁会遇到：** 用户和管理员几乎同时操作同一条申领时
**修复难度：** 简单——删除时加 `.eq('status', 'pending')` 并检查返回行数

---

### 6. 采购审批的并发竞态——最后写入者获胜

**文件：** `src/pages/purchase-approvals/PurchaseApprovalReview.tsx:94-121`

**问题：** 审批操作没有加 `.eq('approval_status', 'pending')` 条件。两个老师同时操作同一条采购申请，一个批准一个拒绝，最后完成的那个操作决定最终结果，前一个操作被静默覆盖。报销审批（`ReimbursementReview.tsx:103-129`）也有同样问题。

**后果：** 审批结果不确定，可能出现「老师 A 批准了但实际被老师 B 的拒绝覆盖」的情况
**谁会遇到：** 多个有审批权限的老师同时处理审批
**修复难度：** 简单——update 加 status 条件并检查返回行数

---

### 7. 文件存储泄漏——多个场景下删除记录不清理云存储文件

**涉及文件：**
- `src/pages/purchase-approvals/PurchaseApprovalList.tsx:443-449`（报销被拒重新提交，旧文件不删）
- `src/pages/reimbursements/ReimbursementList.tsx:98-113`（撤回报销，文件不删）
- `src/pages/purchase-approvals/PurchaseApprovalForm.tsx:300-314`（编辑被拒采购，旧附件不删）
- `src/pages/documents/DocumentEdit.tsx:210-211`（编辑文档移除附件，不删存储）
- `src/lib/storage.ts:17`（`deleteStorageFiles` 不检查返回值，失败静默忽略）

**问题：** 多个操作流程中，数据库记录被更新或删除，但对应的 Supabase Storage 文件没有被清理。`deleteStorageFiles` 函数本身也不检查删除是否成功。

**后果：** 存储空间只增不减，长期运行后可能占满存储配额；被「删除」的文件实际仍可通过 URL 访问（隐私问题）
**谁会遇到：** 所有涉及文件上传的操作——报销、采购、文档、公告
**修复难度：** 中等——需要在每个删除/更新流程中添加存储清理逻辑，并让 `deleteStorageFiles` 正确处理错误

### 8. 上传中断导致孤儿文件

**文件：**
- `src/pages/purchase-approvals/PurchaseApprovalForm.tsx:190-218`
- `src/pages/reimbursements/ReimbursementForm.tsx:93-121`

**问题：** 多文件上传是顺序进行的（for 循环）。如果上传了 3 个文件后第 4 个失败，前 3 个文件已经在 Storage 里了，但因为数据库记录没有创建，这些文件变成永久孤儿——没有任何记录指向它们，也无法找到和清理。

**后果：** 存储空间被无法追踪的文件占用
**谁会遇到：** 网络不稳定时上传多个文件的用户
**修复难度：** 中等——需要实现失败回滚（删除已上传的文件）

### 9. 药品编辑表单绕过库存变动追踪

**文件：** `src/pages/reagents/ReagentForm.tsx:292, 543-549`

**问题：** 编辑药品时可以直接修改 `stock` 字段（比如从 10 改成 50），但不会生成 `reagent_stock_movements` 记录。这导致库存变动日志与实际库存不一致。

**后果：** 库存变动日志无法追溯所有库存变化，盘库时对不上
**谁会遇到：** 管理员通过编辑表单调整药品库存
**修复难度：** 简单——编辑时检测 stock 变化，自动插入变动记录

### 10. 报销可以不传任何凭证就提交

## 二、🟡 用户体验差但不致命的问题

### 11. 登录后不会回到之前想访问的页面

**文件：** `src/components/ProtectedRoute.tsx:33`、`src/pages/auth/LoginPage.tsx:92`

**问题：** 未登录用户访问 `/supplies/reserve` 会被跳转到 `/login`，但登录后固定跳回首页 `/`，不会回到原来想去的页面。分享链接或书签都会有这个问题。

**谁会遇到：** 所有从链接或书签进入系统的用户

### 12. 报销审批路由权限与导航不一致

**文件：** `src/App.tsx:106`、`src/contexts/AuthContext.tsx:163`、`src/components/Layout.tsx:108`

**谁会遇到：** 被分配了报销模块权限但不是 admin 角色的用户
**修复难度：** 简单——路由改为 `requiredModule="reimbursements"`

### 14. 两套报销系统并存，用户容易混淆

**文件：**
- `src/pages/reimbursements/ReimbursementList.tsx`（查询旧的 `reimbursements` 表）
- `src/pages/reimbursements/ReimbursementForm.tsx`（写入新的 `purchases` 表）
- `src/pages/reimbursements/ReimbursementReview.tsx`（查询 `purchases` 表）

**问题：** 系统中存在两套报销数据：旧的 `reimbursements` 表和新的 `purchases` 表上的报销字段。新提交的报销写入 `purchases` 表，但报销列表页查的是旧的 `reimbursements` 表。新提交的报销在列表页看不到。

**谁会遇到：** 通过新流程提交报销后查看报销列表的用户
**修复难度：** 中等——需要统一报销数据源或在列表页合并两个来源

### 15. 老师无法编辑自己的采购申请

**文件：** `src/pages/purchase-approvals/PurchaseApprovalForm.tsx:129-134`

**问题：** 编辑只允许 `status === 'rejected'` 的记录。但老师的采购申请是自动批准的（auto-approved），永远不会变成 rejected 状态。所以老师提交后发现有笔误，没有任何途径修改，只能撤回重建。

**谁会遇到：** 提交采购后发现信息有误的老师
**修复难度：** 中等——需要设计「已批准也能修改」的流程或允许老师撤回自动批准

### 16. 弹窗点击外部关闭不确认，表单数据直接丢失

**文件：** `src/components/Modal.tsx:36, 49`

**谁会遇到：** 在弹窗内填写表单的所有用户，尤其手机用户
**修复难度：** 中等——需要实现「脏数据检测 + 确认对话」

### 17. 没有分页——所有列表一次性加载全部数据

**涉及文件：**
- `src/pages/reagents/ReagentList.tsx:136-141`

**谁会遇到：** 使用一年以上、数据积累到几百条以上的系统
**修复难度：** 中等——需要实现分页或虚拟滚动

## 四、盲区分析（开发时容易没想到的场景）

| 场景 | 现状 | 影响 |
|------|------|------|
| 学生第一次打开系统 | 没有引导流程或新手提示 | 不知道该先做什么 |
| 手机上填长表单，误触返回 | 没有离开确认，数据全丢 | 重新填写 |
| 提交后等审批结果 | 没有推送通知，得自己反复刷新 | 焦虑+低效 |
| 老师一次收到 20 条审批 | 没有批量操作功能 | 逐条处理非常耗时 |
| 学生毕业了 | 删除用户后关联数据清理不明确 | 历史数据可能丢失或残留 |
| 期末盘库 | 无法导出药品库存清单（只能导使用记录） | 无法对账 |
| 存储空间管理 | 多处文件删除不清理存储，只增不减 | 存储配额逐渐耗尽 |
| 手机上操作弹窗 | 点外面直接关闭，无确认 | 填了半天的内容瞬间消失 |
| 断网后提交 | 没有离线草稿或自动保存 | 数据丢失 |
