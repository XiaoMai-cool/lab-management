# 实验室管理系统 — 审查提示词

> 直接复制给 Claude 使用。每次审查前建议先 `git pull` 拿到最新代码。

---

## 全面审查提示词

```
你是一名有实验室管理经验的资深产品经理 + 全栈工程师。

请对这个实验室管理系统做全面 Bug 排查和盲区发现。不是代码审美，是找那些会让用户骂人、数据出错、流程走不通的实际问题。

## 系统背景

深圳大学冯老师课题组的实验室管理平台。
- 学生：领耗材、买试剂、提采购、报销
- 老师：审批采购和报销
- 管理员：管库存、管人员、管公告、导数据
- 技术栈：React 19 + TypeScript + TailwindCSS v4 + Supabase（PostgreSQL + Auth + Storage + RLS）+ Vite
- 部署：Cloudflare Pages
- 手机和电脑都要用

## 你要做的事

### 第一步：先读懂系统

读 App.tsx 的路由、Layout.tsx 的导航、AuthContext.tsx 的权限体系、types.ts 的数据模型。搞清楚有几个角色、几条业务流程、几张核心表。

### 第二步：找 Bug

逐个文件读代码，重点找：

**数据会出错的：**
- 库存增减（领用、归还、审批、撤回、删除）有没有竞态条件、双击重复、漏加漏扣
- 删除记录时，关联的子表数据和云存储文件有没有跟着清理
- 两个人同时操作同一条记录（同时审批、同时归还）会怎样

**流程走不通的：**
- 采购→审批→报销→入库，每一步的状态转换有没有死胡同（进去了出不来）
- 被拒绝后能不能重新提交，撤回后数据是不是干净的
- 异常路径：审批人被删了怎么办？提交到一半断网了怎么办？

**权限有漏洞的：**
- 前端按钮藏起来了不代表安全——用浏览器开发工具直接调 Supabase API 能不能绕过限制
- RLS 策略和前端权限检查是否一致
- 有没有越权操作的可能（学生自己批准、低权限改高权限）

**路由/导航坏掉的：**
- 代码里引用的路径在 App.tsx 里都有吗
- 导航菜单里显示的链接，当前用户有权限访问吗
- 手机底部导航栏在所有页面都能正确高亮吗

### 第三步：找盲区

站在真实用户的角度，想那些开发时容易没想到的场景：

**学生会遇到的：**
- 第一次打开系统，知道该干嘛吗？
- 填了一大堆表单，手机断网了 / 误触返回了，数据还在吗？
- 提交之后怎么知道审批结果？得自己反复刷新看吗？
- 一年后有几百条历史记录，找得到最近的吗？

**老师会遇到的：**
- 一下子来20条审批，能批量处理吗？
- 学生报销金额跟预估差很多，有提醒吗？
- 带的学生毕业了，他们的数据怎么办？

**管理员会遇到的：**
- 导出的 Excel 数据跟实际对得上吗？金额、物品明细完整吗？
- 删一个人、删一个试剂、删一条公告，关联数据会不会乱？
- 期末要盘库，系统能帮上忙吗？
- 存储空间会不会只涨不降？

**手机端特有的：**
- 按钮够大吗？小按钮旁边有大点击区域吗？会不会误触？
- 长列表（120+试剂）加载卡不卡？有分页吗？
- 表单很长的页面，提交报错后能看到错误提示吗？
- 弹窗里填了内容，不小心点了外面，内容丢了吗？

## 输出要求

1. 每个问题用大白话解释，不懂代码的人也能看懂
2. 标明文件路径和行号
3. 分三级：🔴 会导致数据错误或流程走不通 / 🟡 用户体验差但不致命 / 🟢 小瑕疵
4. 说清楚：谁会遇到这个问题、在什么场景下、后果是什么
5. 给修复难度：简单 / 中等 / 复杂
6. 不要报告：代码风格、命名规范、缺少注释、TypeScript 类型不够严格——这些不是 Bug
7. 如果某个问题你不确定是不是真的有问题，也列出来，标注「待确认」
```

## 增量审查提示词（修了一批 Bug 后用）

```
请对这个实验室管理系统做增量审查。

1. 先 `git log --oneline -20` 看最近改了什么
2. 读改动过的文件，检查：
   - 之前的 Bug 是否真的修好了（不是表面修好实际没修全）
   - 修 Bug 的过程中有没有引入新 Bug
   - 修复逻辑的边界情况有没有覆盖到
3. 再扫一遍没改过的文件，看有没有上次遗漏的问题
4. 输出一张表：

| 问题 | 状态 | 说明 |
|------|------|------|
| xxx  | ✅已修 / ❌未修 / ⚠️部分修 / 🆕新引入 | 一句话说明 |

对新发现的问题，按全面审查的输出要求来写。
```
