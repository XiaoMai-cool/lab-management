# 实验室管理系统 — 产品优化建议 v1

> 评审日期：2026-04-05
> 评审视角：产品经理 + 实验室管理经验
> 系统版本：v2（当前线上版本）

---

## 一、用户角色地图问题

### 1. 角色层级 vs 实际实验室场景的错位

当前5级角色：`super_admin → admin → manager → teacher → student`

**问题：**

- **缺少「研究生/博士生」与「本科生」的区分**。实验室里，研究生往往承担值日排班、耗材管理等「半管理」职责，但当前 student 角色是扁平的，没有区分高年级研究生（可以帮导师审批一些小额采购）和刚进实验室的新生（只能领用）。
- **manager 角色定义模糊**。在高校实验室里，"板块负责人" 通常就是研究生，不是独立的管理层。建议考虑用 `module permission` 直接赋予 student，而不是额外设一个 manager 角色，减少角色膨胀。
- **teacher 角色的审批权限过于统一**。实际场景中，不同老师管不同方向——有的管采购，有的管安全，有的只管自己的学生。当前的 `managed_modules` 机制已经部分解决了，但 `isTeacher` 作为一个粗粒度判断仍被大量使用（比如采购审批），建议进一步细化。

**建议：**

- 考虑将 manager 合并到 student，通过 `managed_modules` 赋能即可
- 增加「学生组长」标签（tag），而非独立角色
- 为 student 增加「年级/入学年份」字段，方便值日排班按年级轮转

### 2. 师生绑定关系太简单

当前 `student_teacher_assignments` 是 1:1 映射（一个学生只能绑定一个老师）。

**实际场景问题：**

- 联合培养的学生可能有两个导师
- 学生可能需要向不同老师申请不同类型的采购（试剂找 A 老师审批，设备找 B 老师审批）
- 老师离职/调岗时，学生的绑定关系需要批量迁移

**建议：**

- 改为 1:N（一个学生可绑定多个老师，标注主/副导师）
- 或者采购审批时允许学生手动选择审批人（当前已支持 override，但默认逻辑仍是 1:1）

## 二、核心功能交互优化

### 3. 首页 Dashboard 信息密度过高

当前首页包含：快捷操作、待办任务、公告、值日、每日提醒、库存预警……

**问题：**

- 对学生来说，大部分管理类信息是噪音
- 对管理员来说，需要的是「需要我处理的事项」，不是全部信息
- 手机端滑动距离太长

**建议：**

- **按角色分流首页**：学生首页只显示「我的待办 + 最近公告 + 今日值日」；管理员首页显示「待审批 + 库存预警 + 系统统计」
- 当前已有角色判断逻辑，但内容模块没有做优先级排序——**把「需要我操作的」放最上面**
- 增加一个「已读/未读」状态给公告，避免反复看到旧公告

### 4. 采购-报销流程链路太深

当前流程：`提交采购 → 老师审批 → 自行购买 → 提交报销 → 管理员审批 → 登记入库`

这是6步流程，分散在多个页面：

```
/purchase-approvals/new → /purchase-approvals/review → (线下购买) →
/reimbursements/new → /reimbursements/review → /purchases/registration
```

**已实现的部分（✅ 无需重复建设）：**

- 学生在 `/purchase-approvals` 已能看到自己所有采购记录，`ApprovalStatusLine` 组件展示审批+报销双阶段状态
- 报销入口已从采购列表的「去报销」按钮跳转（`/reimbursements/new?purchase_id={id}`），独立访问会提示「请从采购列表发起报销」
- 已有筛选按钮：全部 / 进行中 / 已完成 / 已拒绝

**仍存在的问题：**

- 状态展示是文字标签，缺少直观的进度条/Timeline 可视化（当前是两行文字状态，不如一条进度条直观）
- 登记入库页面对学生不可见，学生无法确认"东西到了没"
- ⚠️ **代码 Bug**：被拒绝的采购显示「修改」按钮，跳转到 `/purchase-approvals/edit/{id}`，但此路由未在 `App.tsx` 中定义，是死链接

**建议：**

- 将文字状态升级为**可视化进度条**（申请 → 审批 → 报销 → 入库），用户一眼看到全貌
- 修复「修改」按钮的死链接：实现编辑路由，或拒绝后引导用户重新提交
- 考虑让学生看到登记状态（只读），确认物品是否已入库

### 5. 耗材领用流程可简化

当前流程：`SupplyReserve 页面 → 选物品 → 填数量 → 填用途 → 提交 → 等审批 → 审批通过 → 领取`

**实际实验室场景：**

- 很多常用耗材（手套、枪头、离心管）不需要审批，拿了登记一下就行
- 贵重物品（玻璃仪器）才需要审批
- 频繁的审批会导致管理员疲劳，最终形同虚设

**建议：**

- 区分「免审领用」和「审批领用」：在耗材管理中增加 `require_approval` 字段
- 免审耗材：学生提交后直接扣库存，管理员只需事后查看领用记录
- 审批耗材：保持现有流程
- 增加「快速领用」入口：搜索 → 选数量 → 一键提交（跳过用途选择等非必要字段）

### 6. 化学试剂预警机制可进一步自动化

**已实现的部分（✅）：**

- Dashboard 已有自动低库存检测：查询 `stock <= min_stock` 的耗材和试剂，在首页「库存预警」区域展示
- 化学品预警支持 ordered 状态到达预计日期后自动转为 arrived

**仍存在的问题：**

- 低库存**检测展示**有了，但不会自动**创建 warning 记录**——仍需人工去 ChemicalWarnings 页面手动上报
- 预警状态变更（ordered → arrived）没有通知机制，管理员需要主动去查看
- Dashboard 的低库存提醒是只读展示，没有快捷操作入口

**建议：**

- 当 `stock < min_stock` 且该试剂没有活跃 warning 时，自动创建 warning 记录（或至少提供一键创建按钮）
- 预警状态变更时推送通知（邮件或微信）
- Dashboard 的低库存提醒增加「一键创建采购申请」按钮

## 三、移动端交互优化

### 7. 底部导航栏 Tab 选择

当前5个 Tab：`首页 | 耗材 | 试剂 | 采购 | 我的`

**问题：**

- 「耗材」和「试剂」是两个并列的库存管理入口，对学生来说使用频率差异很大
- 没有「消息/通知」入口——审批结果、预警通知在哪看？
- 「采购」这个 Tab 名字不够直观，实际上是「采购+报销」的混合入口

**建议方案一（通知优先）：**

```
首页 | 库存 | + | 通知 | 我的
```

- 「库存」合并耗材+试剂（用顶部 Tab 切换）
- 「+」是快速操作按钮（领用、采购申请、报销）
- 「通知」展示审批结果、预警、公告

**建议方案二（保持现有，微调）：**

- 「采购」改名为「申请」（更通用）
- 在「我的」页面增加「通知中心」入口
- 首页的 badge 数字提示待处理事项

### 8. 表单填写体验（手机端）

**问题：**

- 采购表单字段多（类别、名称、规格、数量、金额、附件...），手机端需要大量滚动
- 没有草稿保存功能——填到一半误触返回，全部丢失
- 附件上传没有拍照快捷入口（手机场景下拍发票很常见）

**建议：**

- 表单分步骤展示（Step 1: 基本信息 → Step 2: 明细 → Step 3: 附件），每步一屏
- 增加草稿自动保存（localStorage 或 Supabase draft 状态）
- 附件上传增加「拍照」快捷按钮（调用 `capture="camera"`）
- 金额输入增加数字键盘（`inputmode="decimal"`）

### 9. 值日模块交互太弱

当前值日页面 (`/duty`) 只是展示谁值日，交互极少。

**实际场景：**

- 值日人经常需要换班
- 需要知道值日具体要做什么（清洁、锁门、检查设备等）
- 值日完成后需要打卡确认

**建议：**

- 增加「换班申请」功能（A 和 B 协商后互换某天值日）
- 增加值日 checklist（管理员配置，值日人逐项打勾）
- 增加值日完成确认/拍照上传（可选，防止「只挂名不干活」）
- 值日提醒推送（当天早上 / 前一天晚上）

## 四、缺失但高价值的功能

### 10. 通知系统

**这是当前系统最大的缺失。** 所有审批流程都是「提交后等着」，没有主动通知。

- 采购审批通过/驳回 → 学生不知道
- 报销审核完成 → 学生不知道
- 库存预警 → 管理员不知道
- 值日提醒 → 值日人不知道

**建议：**

- Phase 1：站内通知（`notifications` 表 + 首页/底栏红点提示）
- Phase 2：邮件通知（Supabase Edge Functions）
- Phase 3：微信推送（已在规划中的 WeChat 集成）

### 11. 操作日志 / 审计追踪

实验室管理涉及化学品、资金，需要可追溯性。

- 谁在什么时候修改了库存数量？
- 谁删除了某条采购记录？
- 谁修改了某人的权限？

**建议：**

- 增加 `audit_logs` 表，记录关键操作（库存变动、权限变更、审批操作）
- 管理员可查看操作日志

### 12. 数据看板 / 统计分析

当前只有报销统计 (`/reimbursements/stats`)，其他模块没有数据分析。

**建议增加：**

- 耗材消耗趋势图（哪些耗材用得最快，预测何时需要采购）
- 试剂使用频率排行
- 各成员采购/报销金额统计
- 月度/季度实验室运营报告自动生成

### 13. 搜索体验

当前各模块有独立搜索，但没有全局搜索。

**建议：**

- 增加全局搜索（Command+K / 顶部搜索框）
- 搜索范围：耗材名称、试剂名称、采购记录、文档标题、人员姓名
- 搜索结果分类展示，点击直接跳转

## 五、优先级排序

| 优先级 | 优化项 | 影响面 | 工作量 | 备注 |
|--------|--------|--------|--------|------|
| **P0** | 通知系统（站内） | 所有用户 | 中 | 系统最大缺失，审批流程依赖 |
| **P0** | 采购进度条/Timeline | 所有用户 | 小 | 提升流程透明度 |
| **P1** | 表单草稿保存 | 所有用户 | 小 | 防止手机端误操作丢失数据 |
| **P1** | 耗材免审/审批分级 | 所有用户 | 中 | 降低管理员审批疲劳 |
| **P1** | 首页按角色分流 | 所有用户 | 小 | 减少信息噪音 |
| **P2** | 底部导航栏优化 | 手机用户 | 中 | 涉及信息架构调整 |
| **P2** | 值日换班 + Checklist | 值日人员 | 中 | 提升值日管理实用性 |
| **P2** | 试剂自动预警 | 试剂管理员 | 小 | 替代手动上报 |
| **P3** | 全局搜索 | 所有用户 | 中 | 提升查找效率 |
| **P3** | 操作审计日志 | 管理员 | 中 | 合规与追溯需求 |
| **P3** | 数据看板 | 管理员 | 大 | 辅助决策 |
| **P3** | 师生 1:N 绑定 | 联合培养学生 | 小 | 当前 1:1 基本够用 |

## 六、总结

当前系统已经覆盖了实验室管理的核心场景（库存、采购、报销、值日、文档），功能完整度较高。主要优化方向集中在三个层面：

1. **信息流通**：通知系统是最关键的缺失，没有通知的审批流程 = 被动等待
2. **流程体验**：采购链路可视化、耗材分级审批、表单草稿保存
3. **移动体验**：导航结构、表单分步、拍照上传等细节打磨

建议按 P0 → P1 → P2 → P3 的顺序逐步迭代。
