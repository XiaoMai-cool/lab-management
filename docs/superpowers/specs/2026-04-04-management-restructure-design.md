# 管理功能重构设计

## 背景

当前手机端管理面板将所有管理入口平铺展示（最多9个），缺乏逻辑分组，用户难以快速找到目标功能。同时，采购审批和报销是两个独立模块，学生需要分别提交，无法看到一条采购从申请到报销的完整流程。

## 改动范围

两个独立但相关的改动：

1. **管理面板分组** — 纯前端改动
2. **采购流程合并** — 前后端改动

---

## 一、管理面板分组

### 现状

`Layout.tsx` 中的 `getManageItems()` 返回一个扁平列表，手机端以3列 grid 平铺展示，所有入口视觉上无区分。

### 目标

按职责将管理功能分组，分组标题 + 子功能直接平铺，打开面板即可一览全部内容，零额外点击。每个分组用不同颜色区分。用户只看到自己有权限的分组。

### 分组结构

| 分组 | 颜色 | 包含功能 | 可见角色 |
|-----|------|---------|---------|
| 采购审批（独立入口） | 绿色 | 审批学生采购申请 | 教师（isTeacher） |
| 报销管理 | 黄色 | 报销审批、报销统计 | 报销审批人（isReimbursementApprover）、耗材专人（isSuppliesManager）、药品专人（isChemicalsManager）。注：耗材/药品专人在此分组中只看到「报销统计」，看不到「报销审批」 |
| 耗材管理 | 蓝色 | 申领审批、库存管理、借用管理、入库登记（非药品） | 耗材专人（isSuppliesManager） |
| 药品管理 | 紫色 | 药品库存、补货管理、入库登记（药品） | 药品专人（isChemicalsManager） |
| 系统管理 | 灰色 | 公告管理、人员管理（含学生-教师对应关系配置）、数据导出 | 管理员（isAdmin） |

### 实现方式

修改 `Layout.tsx`：

- 将 `getManageItems()` 改为返回分组结构（`ManageGroup[]`），每个分组包含 `label`、`color`、`items`
- 采购审批不归入任何分组，作为独立入口渲染在分组列表上方
- 手机端管理面板：遍历分组，每个分组渲染标题 + 3列 grid 子功能
- 桌面端侧边栏：同样按分组渲染，分组标题作为小标题分隔

### 数据结构

```typescript
interface ManageGroup {
  label: string;
  colorScheme: 'green' | 'yellow' | 'blue' | 'purple' | 'gray';
  items: NavItem[];
}

// 采购审批独立于分组之外
interface ManageConfig {
  standalone: NavItem[];  // 采购审批
  groups: ManageGroup[];  // 报销、耗材、药品、系统
}
```

## 二、采购流程合并

### 现状

- 学生在「采购审批」模块提交采购申请，教师审批
- 审批通过后，学生在「报销」模块重新提交报销申请，报销审批人审批
- 两个模块独立，数据通过 `purchase_approval_id` 弱关联
- 报销通过后，无自动流转到入库登记环节

### 目标

将采购审批和报销合并为一条记录，学生只需提交一次，后续所有环节在同一条记录上流转。

### 类别体系

扩充后的采购类别列表：

| 类别 | 说明 | 默认入库 |
|-----|------|---------|
| 试剂药品 | 化学/生物试剂、标准品、培养基 | 是 |
| 实验耗材 | 一次性耗材、玻璃器皿、样品瓶等 | 是 |
| 设备配件 | 仪器维修、零配件、升级改造 | 是 |
| 服装劳保 | 实验服、防护用品 | 是 |
| 测试加工 | 外送检测、样品加工 | 否 |
| 会议培训 | 会议费、培训费 | 否 |
| 出版知产 | 版面费、专利费、文献数据库 | 否 |
| 办公用品 | 打印、文具、办公耗材 | 否 |
| 差旅交通 | 出差、交通费 | 否 |
| 邮寄物流 | 快递、运输费 | 否 |
| 其他 | 以上未覆盖的支出 | 否 |

选择类别时，系统根据上表自动设置「无需入库」开关的默认值，用户可手动修改。

### 流程设计

#### 学生视角

一条采购记录的完整流程（默认）：

```
待审批 → 已批准（去买东西）→ 待报销（上传发票）→ 已报销
```

所有采购均需经过小老师审批，提交表单包含一个开关（根据类别自动设置默认值）：
- **无需入库**：勾选后跳过入库环节，不流转给耗材/药品专人

- 学生/老师提交采购申请（标题、金额、类别、采购类型、描述、附件）
- 教师审批通过后，记录进入「已批准」状态
- 学生采购完成后，在同一条记录上补充报销信息（发票、实际金额等）
- 报销审批人审批报销
- 学生全程只看到「审批」和「报销」两个阶段，不涉及入库

**驳回处理**：
- 审批被驳回：学生可修改申请内容后重新提交，无需新建记录
- 报销被驳回：学生可修改报销信息（发票、金额等）后重新提交

#### 教师（小老师）视角

- 审批学生的采购申请（待审批 → 已批准/已驳回）
- 审批时可修改「无需入库」选项
- 审批后记录**持续可见**，能看到后续报销状态（未报销/已报销）
- 可查看自己名下所有学生的采购记录历史，方便溯源
- 学生之间数据隔离（学生只能看到自己的记录），但小老师能看到分配给自己的所有学生记录
- 审批列表中**区分学生申请和老师申请**，方便识别来源

#### 老师自行采购

- 老师（教师角色）提交采购申请时，**审批自动通过**，记录直接进入「已批准」状态
- 报销流程正常走李健楠审批，与学生一致
- 老师的采购记录在自己的采购审批「已处理」列表中可见，方便数据溯源

#### 学生选择审批教师

- 学生提交采购申请时需选择审批教师
- 已配置对应关系时：系统**自动默认选中**对应的老师，学生也可手动切换
- 未配置对应关系时：不做默认选择，学生手动选择老师

#### 管理员视角

审批通过后，报销和入库是两个独立环节，不强制先后顺序：

- **报销审批人（李健楠）**：看到待报销的记录，审批时可看到该记录的入库状态（已登记/未登记）
- **耗材专人（王子寒）/ 药品专人（宋艳芳）**：审批通过后即可看到需要入库的记录（`skip_registration = false`），登记时可看到该记录的报销状态（已报销/未报销）

状态互通只显示轻量标签，不展示对方环节的详细信息。

### 入库路由规则

根据 `category` 自动分配入库负责人：

| 路由目标 | category |
|---------|----------|
| 药品专人（宋艳芳） | 试剂药品 |
| 耗材专人（王子寒） | 实验耗材、设备配件、服装劳保 |
| 无需入库（默认） | 测试加工、会议培训、出版知产、办公用品、差旅交通、邮寄物流、其他 |

注：「无需入库」仅为默认值，用户可手动修改。若用户将默认无需入库的类别改为需要入库，则按非药品路由至耗材专人。

### 数据模型变更

将 `purchase_approvals` 和 `reimbursements` 表合并为统一的 `purchases` 表。旧表数据为测试数据，无需迁移，直接建新表，旧表废弃。

```
purchases:
  id
  applicant_id        -- 申请人
  title               -- 采购标题
  purchase_type       -- 采购类型：personal(个人采购) / public(公共采购)
  category            -- 类别（见类别体系表）
  estimated_amount    -- 预估金额
  description         -- 描述
  attachments         -- 申请附件

  # 审批阶段
  approver_id         -- 审批教师（老师自行采购时为自己的 ID）
  approval_status     -- pending / approved / rejected
  approval_note       -- 审批备注
  approved_at         -- 审批时间
  auto_approved       -- boolean, 老师自行采购时为 true

  # 流程控制
  skip_registration   -- boolean, 无需入库开关（根据 category 自动设置默认值，可手动修改）

  # 报销阶段
  actual_amount       -- 实际金额
  receipt_attachments -- 发票/收据附件
  reimbursement_status -- null(未提交) / pending / approved / rejected
  reimbursement_reviewer_id -- 报销审批人（系统自动指定，当前固定为李健楠，无需学生选择）
  reimbursement_note  -- 报销备注
  reimbursed_at       -- 报销通过时间

  # 入库阶段（学生不可见）
  registration_status -- null(未登记) / registered
  registered_by       -- 登记人
  registered_at       -- 登记时间

  created_at
  updated_at
```

需要新增学生-教师对应关系（一对一）：

```
student_teacher_assignments:
  id
  student_id          -- 学生 ID（唯一）
  teacher_id          -- 对应教师 ID
  created_at
  updated_at
```

此表在「人员管理」页面中配置。未配置对应关系的学生，提交采购申请时教师下拉菜单不做默认选择，需手动选择。

### 入库登记页面

新增入库登记功能，**一个页面**，根据角色自动过滤数据。在管理面板中以两个入口呈现：

- **非药品入库** → 放在「耗材管理」分组，耗材专人可见，显示实验耗材/设备配件/服装劳保类别
- **药品入库** → 放在「药品管理」分组，药品专人可见，显示试剂药品类别

两个入口指向同一个页面，通过路由参数或角色自动过滤数据。页面展示审批通过且 `skip_registration = false` 的采购记录，管理员可标记「已登记」，同时可看到该记录的报销状态（已报销/未报销）。

### 报销统计页面增强

报销统计只在「报销管理」分组中出现一次，页面内通过 Tab 切换大类：

**Tab 设计**：

| Tab | 包含的 category |
|-----|----------------|
| 全部 | 所有 |
| 试剂药品 | 试剂药品 |
| 实验耗材 | 实验耗材、设备配件、服装劳保 |
| 服务费用 | 测试加工、会议培训、出版知产 |
| 行政开支 | 办公用品、差旅交通、邮寄物流 |
| 其他 | 其他 |

注：所有有权限的角色看到的数据范围相同（全量），Tab 仅用于方便筛选查看，不做权限隔离。

**筛选功能增强**：
- 按姓名筛选
- 按时间范围筛选
- 按金额范围筛选（支持自定义范围，如 200-1000）
- 按类别筛选
- 按采购类型筛选（个人采购 / 公共采购）

### 页面变更

| 当前页面 | 变更 |
|---------|------|
| `PurchaseApprovalForm.tsx` | 改为统一的采购申请表单。增加采购类型、扩充后的类别选择、无需入库开关（根据类别自动默认）。老师提交时审批自动通过。默认选中对应教师，可手动切换 |
| `PurchaseApprovalList.tsx` | 改为学生/老师的采购记录列表，显示完整流程线（审批→报销）。区分学生和老师的记录 |
| `PurchaseApprovalReview.tsx` | 保留教师审批功能，可修改「无需入库」选项，增加已审批记录的后续状态展示（报销状态），区分学生/老师申请 |
| `ReimbursementForm.tsx` | 改为在已批准的采购记录上补充报销信息 |
| `ReimbursementList.tsx` | 废弃，合并到采购记录列表 |
| `ReimbursementReview.tsx` | 保留，报销审批人审批报销 |
| `ReimbursementStats.tsx` | 改为6个 Tab（全部/试剂药品/实验耗材/服务费用/行政开支/其他）+ 增强筛选（姓名、时间范围、金额范围、采购类型） |
| 新增：入库登记页面 | 耗材/药品专人各自查看待登记物资，标记已登记，可看报销状态 |

### 底部导航调整

当前底部导航「报销」Tab 改为「采购」，路由指向统一的采购记录列表。学生在这里完成从申请到报销的全部操作。

### Dashboard 首页调整

- 快捷入口：将「采购审批」和「报销申请」合并为一个「采购申请」入口
- 待办事项：统一展示采购记录的待处理状态，不再分开展示采购和报销

## 三、安全加固

### 现状

- 报销记录：后端 RLS 严格隔离 ✅
- 采购审批、物资申领：RLS 策略为 `SELECT ... USING (true)`，所有登录用户可读全部记录，仅靠前端过滤 ⚠️

### 改进

为 `purchases` 表（合并后）设置严格的 RLS 策略：

```sql
CREATE POLICY "purchases_select" ON purchases
  FOR SELECT TO authenticated USING (
    applicant_id = auth.uid()          -- 申请人看自己的
    OR approver_id = auth.uid()        -- 审批教师看分配给自己的
    OR manages_module('reimbursements') -- 报销审批人
    OR manages_module('supplies')       -- 耗材专人（入库）
    OR manages_module('chemicals')      -- 药品专人（入库）
    OR is_admin_or_above()             -- 管理员
  );
```

同时补齐 `supply_reservations` 表的 RLS 策略。

## 不在范围内

- 采购记录与耗材/药品库存系统的深度集成（如入库时自动创建库存条目）
- 通知系统（推送、站内消息等）
- 桌面端的大幅 UI 重构（仅跟随分组逻辑调整侧边栏）
- 旧表数据迁移（现有数据为测试数据，直接建新表）
